# ModelMate 11월 프로젝트 로드맵

## 최종 목표

ModelMate를 고정 샘플 데모가 아니라, 임의 CSV를 즉석 업로드해도 데이터의 성격을 판단하고 예측 문제를 구성한 뒤 모델 비교, 이유 설명, 저장, 재사용, 공유/API까지 이어지는 범용 AutoML SaaS처럼 보이게 만든다.

## 현재 진행률

- 중간 발표 가능 상태: 85~90%.
- 11월 최종 발표 기준: 55~60%.
- 실제 상용 MVP 느낌 기준: 40~45%.
- 지금까지는 업로드 판단, 도메인/타겟 추론, 모델 비교, 결과 설명, 에이전트 판단, 작업공간 연속성, 실험 재열기, 저장 모델 요약, 공유/API 미리보기, 업로드 QA를 만들었다.
- 앞으로는 계정별 영구 저장, 실험 재사용, 저장 모델 관리, 실제 권한/API, 장기 QA가 핵심이다.

## 월별 기준선

### 6월: 안정 버전

- 중간 발표 가능한 안정 버전을 만든다.
- 업로드, 자동 분석, 모델 선택, 결과 요약, 이유 보기, 저장/재사용 기본 흐름을 안정화한다.
- 여러 CSV를 넣어 깨지는 부분을 줄인다.
- Railway/GitHub 반영 상태를 매 작업 후 확인한다.

### 7월: 작업공간/실험 관리

- 사용자별 데이터셋과 실험 기록을 관리한다.
- 과거 작업을 다시 열고 같은 설정으로 재분석할 수 있게 한다.
- 관리자 화면에서 사용자, 데이터셋, 실험, 실패 흐름을 확인한다.
- 저장 데이터셋, 실험, 보고서, 저장 모델의 연결을 명확히 보여준다.

### 8월: 모델 관리/재사용

- 최고 모델을 이름과 버전으로 저장한다.
- 저장 모델로 새 데이터 예측을 실행한다.
- 모델별 성능, 상태, 사용 가능 여부를 비교할 수 있게 한다.
- 모델 생명주기 상태를 준비, 사용 가능, 검토 필요, 폐기 후보처럼 구분한다.

### 9월: 공유/API/상용화 흐름

- 보고서 공유 링크를 정리한다.
- 공개/비공개 상태를 표시한다.
- API 키 형태의 UX와 사용 예시를 강화한다.
- 실제 운영 기능과 미리보기 기능을 명확히 구분한다.
- 공유/API 사용량, 접근 권한, 보안 안내를 서비스처럼 보이게 만든다.

### 10월: UI/QA/시연 완성도

- 긴 화면을 사이드 패널로 정리한다.
- 모바일/좁은 화면에서도 카드, 버튼, 표, 차트가 깨지지 않게 한다.
- 발표용 시나리오와 백업 데이터셋을 준비한다.
- 실패 케이스도 친절한 안내로 보여준다.

### 11월: 최종 방어

- 발표 흐름을 5~7분 안에 안정적으로 재현한다.
- 임의 CSV를 넣어도 서비스가 판단하고 다음 행동을 제시하는 장면을 보여준다.
- 한계와 향후 확장 계획을 기술적으로 설명할 수 있게 정리한다.

- compact/mobile 줄바꿈 보강.

## 남은 핵심 리스크

- 로그인은 데모 수준이다. 실제 상용화를 주장하려면 계정별 DB 저장과 권한 처리가 필요하다.
- API 키 UX는 미리보기다. 실제 보안 키와 인증 서버 구현은 아직 아니다.
- 저장 모델은 관리 흐름을 보여주는 수준이다. 실제 모델 artifact 저장, 버전 관리, 재학습 이력은 더 필요하다.
- QA는 no-token 자동 검증 중심이다. 실제 브라우저 조작, 모바일 화면, 다양한 공개 CSV를 더 반복해야 한다.
- 임의 CSV 판단은 규칙 기반과 모델 결과를 함께 쓰는 구조다. 완벽한 의미 이해가 아니라 발표 가능한 범용 판단에 초점을 둔다.

## 다음 작업 후보

1. 저장 실험을 열었을 때 타겟, 제외 컬럼, 선택 모델, 점수, 보고서가 완전히 복원되게 만든다.
2. 저장 모델 상세 화면을 추가해 모델 이름, 버전, 상태, 학습 데이터, 타겟, 점수, 사용 가능 여부를 보여준다.
3. 새 데이터 예측 화면에서 저장 모델을 선택해 예측하는 흐름을 강화한다.
4. 공유/API 화면에 공개/비공개 토글, API 키 미리보기, 사용 예시, 주의 문구를 더 명확히 넣는다.
5. 업로드 실패/애매한 CSV에서 “왜 분석하기 어려운지”와 “어떻게 고치면 되는지”를 더 구체화한다.
6. 모바일/좁은 화면에서 report, history, deploy, admin 화면을 다시 점검한다.

## 자동화 지시 기준

다른 Codex 환경에서 이어받으면 우선 `CODEX_HANDOFF.md`를 읽고, `git status --short`, `git log --oneline -8`, Railway URL bundle 확인을 먼저 수행한다. 그 후 위 진행 순서대로 낮은 위험의 작은 단위 작업을 진행하고, 빌드/QA/커밋/푸시/배포 확인을 반복한다.
